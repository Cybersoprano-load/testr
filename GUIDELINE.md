# GUIDELINE — как это всё работает

Документ для понимания стенда: что из чего состоит, как запускать, куда смотреть и что
означают метрики. Читается сверху вниз.

---

## 1. Карта системы и поток данных

```
        ┌─────────┐   produce    ┌──────────┐   consume    ┌───────────────┐   insert   ┌────────────┐
        │ JMeter  │ ───────────▶ │  Kafka   │ ───────────▶ │   Заглушка     │ ─────────▶ │ PostgreSQL │
        │ (.jmx)  │  топик       │ +Zookeeper│  многопоточно│ (Spring Boot)  │  запись    │  messages  │
        └────┬────┘  messages    └────┬─────┘              └───────┬───────┘            └─────┬──────┘
             │                        │                            │                          │
   метрики   │ InfluxDB         kafka-exporter           /actuator/prometheus          postgres-exporter
             ▼                        ▼                            ▼                          ▼
        ┌─────────────────────────────────────── Prometheus ───────────────────────────────────────┐
        │  (InfluxDB — отдельный источник для JMeter)                                                │
        └───────────────────────────────────────────┬──────────────────────────────────────────────┘
                                                     ▼
                                                 ┌────────┐
                                                 │ Grafana│  4 дашборда: JVM, PostgreSQL, Kafka, JMeter
                                                 └────────┘
```

Идея: JMeter создаёт нагрузку (сообщения в Kafka) → заглушка их вычитывает и пишет в БД →
всё это наблюдаем в Grafana. Управляемая задержка в заглушке позволяет искусственно «тормозить»
запись и смотреть, как растёт отставание (lag) и меняются метрики.

Адреса всех сервисов — в таблице в [README.md](README.md).

## 2. Запуск по шагам

1. **Поднять стек:**
   ```bash
   docker compose up -d --build
   ```
   Что происходит: стартуют Zookeeper → Kafka → (создаётся топик `messages`) → PostgreSQL
   (выполняется `init.sql`, создаётся таблица) → собирается и стартует заглушка → поднимаются
   Prometheus, Grafana, InfluxDB и экспортеры.

2. **Проверить, что всё поднялось:**
   ```bash
   docker compose ps
   ```
   У `kafka`, `postgres`, `stub` должно быть `(healthy)`.

3. **Убедиться, что топик и таблица на месте:**
   ```bash
   docker compose exec kafka kafka-topics --bootstrap-server kafka:29092 --describe --topic messages
   docker compose exec postgres psql -U kafka_stub -d kafka_stub -c "\d messages"
   ```

4. **Подать нагрузку** (см. раздел 4).

## 3. Где что смотреть

### Сообщения в Kafka
- **Kafka Tool / Offset Explorer** (desktop):
  1. Add New Connection → вкладка *Properties*.
  2. Cluster name: любое (например `local`).
  3. *Bootstrap servers*: `localhost:9092` (тип кластера без аутентификации, Kafka ≥ 0.11).
  4. Connect → раскрыть Topics → `messages` → вкладка *Data* (поставь Value-формат = String),
     там видно тело сообщений и распределение по 6 партициям.
- **Kafka UI (браузер):** http://localhost:8081 → Topics → `messages` → Messages.

### Данные в БД
- **DBeaver:** подключение `kafka_stub (docker)` (я его уже добавил), `localhost:5432`,
  база/логин/пароль = `kafka_stub`. Таблица `public.messages`.
  > После добавления подключения DBeaver нужно полностью закрыть (Cmd+Q) и открыть заново.
- Или из консоли:
  ```bash
  docker compose exec postgres psql -U kafka_stub -d kafka_stub -c "select * from messages order by id desc limit 10;"
  ```

### Логи заглушки
```bash
docker compose logs -f stub
```
На каждое сообщение две строки в формате задания:
```
2026-06-07 15:52:19.434 – [Read from Kafka] {"msg_uuid": "...", "head": true, "method": "POST", "uri": "/post-message"}
2026-06-07 15:52:19.937 – [Write to DB] { "msgUuid": "...", "head": true, "timeRq": "1780847539" }
```

### Метрики
- **Grafana:** http://localhost:3000 (admin / admin) → Dashboards. Четыре дашборда (см. раздел 6).
- Сырые метрики: `curl localhost:8080/actuator/prometheus`, `curl localhost:9187/metrics`,
  `curl localhost:9308/metrics`, цели Prometheus — http://localhost:9090/targets.

## 4. Как подавать нагрузку (JMeter, п.1.2)

Скрипт: [jmeter/kafka-load-test.jmx](jmeter/kafka-load-test.jmx). Запуск — открыть его в JMeter
(GUI) и нажать Start, либо из консоли:
```bash
jmeter -n -t jmeter/kafka-load-test.jmx -l results.jtl   # 4 ступени по 5 минут (5/10/12/14 оп/с), ≈20 минут
```

Из чего состоит сценарий:
- **Kafka Producer Config** (плагин di-kafkameter) — один продюсер на весь тест, брокер `localhost:9092`.
- **Ultimate Thread Group** — ступени по числу потоков **5 → 10 → 12 → 14**, удержание 5 минут каждая.
- **Constant Throughput Timer** — режим «this thread only», 60 запросов/мин = **1 оп/с на поток**.
  Поэтому суммарный темп = числу активных потоков → ровно **5/10/12/14 оп/с** по ступеням.
- **JSR223 PreProcessor** — на каждое сообщение генерирует случайный `msg_uuid` и тело. Здесь же
  лениво (один раз, потокобезопасно через `putIfAbsent`) заводится глобальный счётчик сообщений,
  по которому **каждое 10-е сообщение получает `head=false`** — едино для всех потоков.
- **Kafka Producer Sampler** — отправляет тело в топик `messages`.
- **Backend Listener (InfluxDB)** — шлёт метрики прогона в InfluxDB → Grafana.

> Темп задаётся именно числом потоков (а не одним Constant Throughput Timer на всю группу),
> потому что ступени задания неравномерные (5,10,12,14). Ultimate Thread Group умеет такие
> ступени, а Stepping Thread Group — только равномерные.

Построчный разбор скрипта генерации сообщения и понятие атомарности — в отдельном файле
[jmeter/JMETER-GUIDE.md](jmeter/JMETER-GUIDE.md).

## 5. Задержка записи (п.3)

- Перед записью в БД заглушка ждёт `delayMs` миллисекунд (по умолчанию 1000).
- Значение хранится в `AtomicLong` и читается на КАЖДОЕ сообщение, поэтому меняется **на лету**:
  ```bash
  curl localhost:8080/api/delay                       # текущее значение
  curl -X PUT localhost:8080/api/delay \
       -H 'Content-Type: application/json' -d '{"delayMs":3000}'
  ```
  После этого в логах видно, что промежуток между `[Read from Kafka]` и `[Write to DB]` стал ~3 c,
  без перезапуска заглушки.
- Текущая задержка также публикуется как метрика `stub_delay_ms` (видно в Prometheus/Grafana).
- Эффект под нагрузкой: чем больше задержка, тем медленнее заглушка разбирает топик → растёт
  **consumer lag** (см. метрики Kafka).

## 6. Метрики — подробно, сгруппированы по заданию (п.2)

В задании четыре группы метрик. По каждой группе ниже: откуда берётся, **какие метрики
основные** (за чем следить в первую очередь), что каждая означает и **какие проблемы**
сигнализирует тревожное значение.

### 6.1. JVM-метрики заглушки
**Источник:** Spring Boot Actuator + Micrometer → `/actuator/prometheus` → Prometheus.
**Дашборд:** «JVM (Micrometer)» (Grafana ID 4701).

**Основные (в первую очередь):** heap-память, паузы GC, число потоков, загрузка CPU.

- **`jvm_memory_used_bytes` / `jvm_memory_max_bytes`** — память по пулам: heap (Eden, Survivor,
  Old Gen) и non-heap (Metaspace, Code Cache).
  - *Что означает:* сколько памяти JVM реально занято и каков потолок.
  - *⚠ Проблемы:* Old Gen после сборок не падает, а ползёт вверх → **утечка памяти**, впереди
    `OutOfMemoryError`. `used` почти равен `max` → частые Full GC и риск OOM.
- **`jvm_gc_pause_seconds`** (count / sum / max) — частота и длительность пауз сборки мусора
  (в эти моменты приложение «замирает»).
  - *Что означает:* сколько времени теряется на GC.
  - *⚠ Проблемы:* частые или длинные паузы (сотни мс) → просадки и рост времени обработки
    сообщений; учащение Full GC → давление на heap (см. память).
- **`jvm_threads_live_threads`, `jvm_threads_states_threads{state=…}`** — сколько потоков живо и
  в каких состояниях (runnable / blocked / waiting). Включает consumer-потоки заглушки и Tomcat.
  - *Что означает:* загруженность пула потоков.
  - *⚠ Проблемы:* непрерывный рост `live` → **утечка потоков**; много `blocked` → конкуренция за
    блокировки/общий ресурс, потоки простаивают.
- **`process_cpu_usage` / `system_cpu_usage`** (доля 0…1) — загрузка CPU процессом и всей системой.
  - *Что означает:* упираемся ли в процессор.
  - *⚠ Проблемы:* процесс у 1.0 (100%) → CPU-bound, throughput выше не поднять без оптимизации;
    высокий `system` при низком `process` → CPU «съедает» что-то другое на хосте.
- **`jvm_classes_loaded_classes`, `process_open_fds`, `process_uptime_seconds`** — загруженные
  классы, открытые файловые дескрипторы, аптайм.
  - *Что означает:* общий «пульс» JVM.
  - *⚠ Проблемы:* дескрипторы растут без предела → утечка сокетов/файлов (упор в лимит ОС);
    внезапный сброс `uptime` → заглушка перезапускалась/падала.
- **`stub_delay_ms`** (наша метрика) — текущая задержка записи в БД.
  - *Что означает:* удобно накладывать на графики lag/throughput и видеть эффект её изменения.

### 6.2. Метрики PostgreSQL
**Источник:** `postgres-exporter` (`:9187/metrics`) → Prometheus.
**Дашборд:** «PostgreSQL Database» (Grafana ID 9628).

**Основные (в первую очередь):** число соединений, TPS (commits), cache hit ratio, `pg_up`.

- **`pg_stat_activity_count`** — число соединений к БД (активные + простаивающие). Заглушка
  держит пул HikariCP.
  - *Что означает:* сколько коннектов открыто.
  - *⚠ Проблемы:* упёрлись в `max_connections` → новые запросы ждут/падают с ошибкой
    «too many clients»; много `idle in transaction` → подвисшие незакрытые транзакции.
- **`pg_stat_database_xact_commit` / `pg_stat_database_xact_rollback`** — число коммитов и
  откатов транзакций (берём rate в секунду).
  - *Что означает:* фактическая скорость записи в БД (TPS).
  - *⚠ Проблемы:* commits не растут при росте нагрузки → БД стала узким местом; ненулевые
    `rollback` → ошибки/откаты при вставке.
- **`pg_stat_database_blks_hit` / `pg_stat_database_blks_read`** — чтения из кэша (shared buffers)
  против чтений с диска. Cache hit ratio = hit / (hit + read).
  - *Что означает:* насколько эффективно работает кэш БД.
  - *⚠ Проблемы:* hit ratio заметно ниже ~0.99 → много чтений с диска, дисковая подсистема
    тормозит запросы.
- **`pg_database_size_bytes`** — размер базы (и косвенно рост таблицы `messages`).
  - *Что означает:* сколько данных накопилось по ходу нагрузки.
  - *⚠ Проблемы:* неконтролируемый рост → закончится место на диске.
- **`pg_stat_database_deadlocks`, `..._temp_bytes`, `..._tup_inserted`** — взаимоблокировки,
  объём временных файлов, число вставленных строк.
  - *Что означает:* аномалии и реальный объём вставок (tup_inserted должен расти вместе с нагрузкой).
  - *⚠ Проблемы:* ненулевые `deadlocks` → конкурирующие транзакции блокируют друг друга; рост
    `temp_bytes` → запросам не хватает `work_mem`, сортировки уходят на диск.
- **`pg_up`** — доступность БД/экспортера (1 = ок).
  - *⚠ Проблемы:* `0` → экспортер не достучался до PostgreSQL (БД упала или недоступна).

### 6.3. Метрики JMeter
**Источник:** Backend Listener в .jmx → InfluxDB (db `jmeter`) → Grafana.
**Дашборд:** «Apache JMeter Dashboard using Core InfluxdbBackendListenerClient» (Grafana ID 5496).

**Основные (в первую очередь):** throughput, % ошибок, перцентили времени отклика.

- **Throughput** (операций/с) — реальный темп отправки сообщений в Kafka.
  - *Что означает:* фактическая нагрузка; должен идти ступенями ~5 → 10 → 12 → 14.
  - *⚠ Проблемы:* фактический темп ниже целевого → JMeter/клиент не успевает (мало потоков,
    упор в CPU генератора или в брокер); «полка» вместо ступеней → брокер не принимает быстрее.
- **`countError` / % ошибок** — доля неуспешных сэмплов.
  - *Что означает:* сколько отправок не удалось.
  - *⚠ Проблемы:* >0 → продюсер не достучался до брокера, таймауты, переполнен буфер
    (`buffer.memory`), недоступен топик. В норме строго 0%.
- **Response time** — avg / min / max и перцентили **pct90 / pct95 / pct99** (для Kafka это время
  отправки сообщения).
  - *Что означает:* задержка операции; перцентили показывают «хвосты» (худшие случаи), а не
    усреднённую картину.
  - *⚠ Проблемы:* растущие pct95/pct99 при росте ступени → брокер/сеть начинают тормозить под
    нагрузкой; большой разрыв avg ↔ pct99 → редкие, но сильные задержки (например, паузы GC брокера).
- **Active threads** — число активных потоков сейчас.
  - *Что означает:* должно повторять ступени 5 → 10 → 12 → 14 (проверка, что профиль отрабатывает).
  - *⚠ Проблемы:* потоков меньше запланированного → группа не успела разогнаться/преждевременно
    завершилась.

### 6.4. Метрики Kafka
**Источник:** `kafka-exporter` (`:9308/metrics`) → Prometheus.
**Дашборд:** «Kafka Exporter Overview» (Grafana ID 7589).

**Основные (в первую очередь):** consumer lag (главная метрика) и скорость роста offset топика.

- **`kafka_consumergroup_lag`** — **ГЛАВНАЯ метрика**: на сколько сообщений группа заглушки
  отстаёт от конца топика (сколько ещё не прочитано), по партициям и суммарно.
  - *Что означает:* справляется ли заглушка с входящим потоком.
  - *⚠ Проблемы:* lag постоянно растёт → заглушка **не успевает** (велика задержка записи, мало
    consumer-потоков, медленная БД); lag «перекошен» по партициям → неравномерное распределение
    или одна партиция/поток тормозит. Норма — lag около нуля или колеблется без устойчивого роста.
- **`kafka_topic_partition_current_offset`** — текущий конечный offset каждой из 6 партиций
  (сколько всего записано).
  - *Что означает:* скорость роста = входящий темп от JMeter (должен отражать ступени).
  - *⚠ Проблемы:* не растёт во время теста → сообщения не доходят до Kafka (проблема продюсера);
    весь рост в одной партиции → плохой ключ распределения, многопоточность не используется.
- **`kafka_consumergroup_current_offset`** — до какого места дочитала группа заглушки.
  - *Что означает:* прогресс вычитки; `lag = offset топика − offset группы`.
  - *⚠ Проблемы:* стоит на месте, а offset топика растёт → консьюмер завис/отвалился от группы.
- **`kafka_topic_partitions`** — число партиций топика (у нас 6).
  - *Что означает:* потолок параллелизма (полезных consumer-потоков не больше, чем партиций).
  - *⚠ Проблемы:* партиций меньше, чем consumer-потоков → лишние потоки простаивают.
- **`kafka_brokers`** — число живых брокеров (у нас 1).
  - *⚠ Проблемы:* стало 0 → брокер упал, весь конвейер встал.

**Главный сценарий наблюдения:** увеличиваем задержку записи (`PUT /api/delay`) → lag растёт
(заглушка не успевает); уменьшаем → заглушка догоняет, lag падает к нулю. Это наглядно
показывает влияние задержки на пропускную способность.
