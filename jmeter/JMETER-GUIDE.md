# JMeter — подробный разбор сценария

Этот файл объясняет, как устроен нагрузочный сценарий [kafka-load-test.jmx](kafka-load-test.jmx):
что такое «атомарность», как построчно работает скрипт генерации сообщения и как все элементы
сценария связаны между собой.

---

## 1. Что такое «атомарный»

**Атомарная операция** — это операция, которая выполняется целиком и неделимо: её нельзя
«перебить» другим потоком на середине. Либо она произошла полностью, либо не произошла вовсе;
промежуточного состояния другие потоки не видят. («Атом» = «неделимый».)

### Зачем это нам
В сценарии одновременно работает до **14 потоков**, и все они должны пользоваться **одним общим
счётчиком сообщений**, чтобы правило «каждое 10-е сообщение `head=false`» считалось по всему тесту,
а не отдельно в каждом потоке.

### Проблема без атомарности — «гонка» (race condition)
Обычное `n = n + 1` — это на самом деле три шага: «прочитать n», «прибавить 1», «записать n».
Если два потока делают это одновременно с общим `n`:

```
n = 5
Поток A: прочитал 5
Поток B: прочитал 5      ← оба увидели одно и то же
Поток A: записал 6
Поток B: записал 6       ← одно увеличение потерялось! Должно было стать 7
```

Итог — два сообщения получили бы один и тот же номер, а счётчик «недосчитал». Правило «каждое
10-е» поехало бы.

### Решение — атомарные инструменты
- **`AtomicLong.incrementAndGet()`** — атомарно увеличивает число и возвращает новое значение.
  Даже если 14 потоков вызовут его одновременно, каждый получит **своё уникальное** значение
  (…, 100, 101, 102 …) без потерь и дубликатов.
- **`props.putIfAbsent(key, value)`** — атомарно «положи значение, только если ключа ещё нет».
  Гарантирует, что счётчик создастся **ровно один раз**, даже если несколько потоков попытаются
  создать его в один момент (`props` в JMeter — это `Hashtable`, у которого `putIfAbsent`
  синхронизирован).

Вместе они дают потокобезопасный общий счётчик без ручных блокировок.

## 2. Скрипт генерации сообщения — построчно

Это код **JSR223 PreProcessor** (язык Groovy). PreProcessor выполняется **перед каждой
отправкой** сообщения, в том же потоке, что и сэмплер. Полный скрипт:

```groovy
import java.util.concurrent.atomic.AtomicLong
import groovy.json.JsonOutput

props.putIfAbsent("msgCounter", new AtomicLong(0))
long n = ((AtomicLong) props.get("msgCounter")).incrementAndGet()
boolean head = (n % 10 != 0)
String uuid = UUID.randomUUID().toString()

vars.put("msg_uuid", uuid)
vars.put("kafkaBody", JsonOutput.toJson([msg_uuid: uuid, head: head, method: "POST", uri: "/post-message"]))
```

Разбор по строкам:

1. `import java.util.concurrent.atomic.AtomicLong` — подключаем класс атомарного счётчика.
2. `import groovy.json.JsonOutput` — подключаем удобный сборщик JSON.
3. `props.putIfAbsent("msgCounter", new AtomicLong(0))`
   — кладём в **общее хранилище `props`** счётчик `msgCounter` со значением 0, **но только если
   его там ещё нет**. Первый поток, дошедший сюда, создаст счётчик; все остальные ничего не
   перезапишут. `new AtomicLong(0)` создаётся при каждом вызове, но «лишний» объект просто
   отбрасывается, если ключ уже занят — это дёшево.
4. `long n = ((AtomicLong) props.get("msgCounter")).incrementAndGet()`
   — достаём тот самый общий счётчик, **атомарно** увеличиваем его на 1 и берём новое значение в
   `n`. Это **порядковый номер текущего сообщения** во всём тесте (1, 2, 3, …). Каждый поток
   получит своё уникальное `n`.
5. `boolean head = (n % 10 != 0)`
   — `n % 10` это остаток от деления на 10. Для n = 1..9 остаток ≠ 0 → `head = true`. Для n = 10
   остаток = 0 → `head = false`. Дальше 11..19 → true, 20 → false, и так далее. То есть **каждое
   10-е сообщение получает `head=false`**, остальные — `true`.
6. `String uuid = UUID.randomUUID().toString()` — генерируем случайный `msg_uuid` для сообщения.
7. `vars.put("msg_uuid", uuid)` — кладём uuid в **переменные потока `vars`**, чтобы сэмплер мог
   взять его как ключ сообщения Kafka (`${msg_uuid}`).
8. `vars.put("kafkaBody", JsonOutput.toJson([...]))` — собираем тело сообщения в JSON и кладём в
   `vars` под именем `kafkaBody`. `JsonOutput.toJson` превращает Groovy-словарь
   `[msg_uuid: …, head: …, method: "POST", uri: "/post-message"]` в строку
   `{"msg_uuid":"…","head":true,"method":"POST","uri":"/post-message"}`. Сэмплер потом возьмёт её
   как `${kafkaBody}`.

### `props` против `vars` — важное различие
| | `props` | `vars` |
|---|---|---|
| Область видимости | **общие** на весь тест, видят все потоки | **локальные** для одного потока |
| Зачем здесь | общий счётчик `msgCounter` (нужен всем) | `msg_uuid` и `kafkaBody` (своё на каждое сообщение) |

Поэтому счётчик лежит в `props` (один на всех), а данные конкретного сообщения — в `vars`
(у каждого потока свои, не мешают друг другу).

### Полный список переменных

| Переменная | Где живёт | Тип | Что это и зачем |
|---|---|---|---|
| `props` | встроенный объект JMeter | `Properties` (общий на весь тест) | глобальное хранилище «ключ→значение», видят все потоки. Сюда кладём общий счётчик. |
| `vars` | встроенный объект JMeter | `JMeterVariables` (свой у каждого потока) | переменные текущего потока. Сюда кладём данные конкретного сообщения. |
| `msgCounter` | ключ внутри `props` | `AtomicLong` | один общий счётчик сообщений на весь тест; атомарно растёт 1,2,3… |
| `n` | локальная в скрипте | `long` | порядковый номер **текущего** сообщения (значение счётчика после +1). |
| `head` | локальная в скрипте | `boolean` | флаг для этого сообщения: `false` у каждого 10-го, иначе `true`. |
| `uuid` | локальная в скрипте | `String` | случайный `msg_uuid`, сгенерированный для сообщения. |
| `msg_uuid` | переменная JMeter (в `vars`) | `String` | тот же uuid; сэмплер берёт его как **ключ** сообщения через `${msg_uuid}`. |
| `kafkaBody` | переменная JMeter (в `vars`) | `String` (JSON) | готовое тело сообщения; сэмплер шлёт его через `${kafkaBody}`. |

Отдельно — переменные уровня всего сценария (не в скрипте):

| Переменная | Где задаётся | Что это |
|---|---|---|
| `KafkaProducerClient` | Kafka Producer Config и Sampler (поле «Variable Name») | имя, под которым хранится созданный продюсер Kafka; **связывает** Config (создаёт продюсер) и Sampler (использует его). |
| `kafka.brokers` | JMeter property, `${__P(kafka.brokers,localhost:9092)}` | адрес брокера; по умолчанию `localhost:9092`, можно переопределить через `-Jkafka.brokers=…`. |
| `kafka.topic` | JMeter property, `${__P(kafka.topic,messages)}` | имя топика; по умолчанию `messages`, переопределяется через `-Jkafka.topic=…`. |

> `${имя}` — это подстановка значения переменной из `vars`. `${__P(имя, по_умолчанию)}` —
> подстановка **property** (значения уровня всего JMeter, заданного через `-J` или дефолт).

## 3. Как скрипт связан с отправкой

PreProcessor только **готовит** переменные. Само сообщение отправляет **Kafka Producer Sampler**,
который ссылается на эти переменные:

- `Kafka Message Key = ${msg_uuid}` — ключ сообщения (влияет на выбор партиции).
- `Kafka Message = ${kafkaBody}` — тело сообщения (тот самый JSON).

Порядок на каждой итерации потока: **PreProcessor (готовит `msg_uuid` и `kafkaBody`) → Sampler
(отправляет) → Constant Throughput Timer (выдерживает паузу до 1 оп/с)**.

---

## 4. Весь сценарий целиком (как работает принцип)

```
Kafka Producer Config        — один раз создаёт продюсер Kafka (брокер localhost:9092)
└─ Ultimate Thread Group     — расписание потоков: 5 → 10 → 12 → 14, каждая ступень 5 минут
   ├─ JSR223 PreProcessor    — готовит msg_uuid + тело, ведёт глобальный счётчик (каждое 10-е head=false)
   ├─ Kafka Producer Sampler — отправляет ${kafkaBody} с ключом ${msg_uuid} в топик messages
   └─ Constant Throughput Timer — 60/мин «this thread only» = 1 оп/с на поток (пейсинг)
Backend Listener (InfluxDB)  — собирает метрики прогона и шлёт в InfluxDB → Grafana
```

Принцип темпа: **темп = число потоков**, потому что каждый поток ограничен 1 операцией в секунду.
- 5 потоков × 1 оп/с = 5 оп/с
- 10 → 10 оп/с, 12 → 12 оп/с, 14 → 14 оп/с

Ultimate Thread Group меняет **число потоков** ступенями (он умеет неравномерные ступени
5/10/12/14), а Constant Throughput Timer держит **темп каждого потока** ровно 1 оп/с. В сумме —
ровно те ступени, что нужны в задании.

## 5. Как запустить
- В GUI: открыть `kafka-load-test.jmx` в JMeter и нажать **Start**.
- Из консоли (без GUI):
  ```bash
  jmeter -n -t kafka-load-test.jmx -l results.jtl
  ```
Метрики прогона идут в InfluxDB (`localhost:8086`, db `jmeter`) и видны в Grafana на дашборде
«Apache JMeter Dashboard…». Что означают метрики JMeter — см. [../GUIDELINE.md](../GUIDELINE.md), раздел 6.3.
